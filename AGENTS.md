# AGENTS.md

## 工作原则

以第一性原理开发：先确认原始需求、状态约束和失败模式，再写实现。

1. 目标不清晰时先停下来讨论，不假设产品意图。
2. 路径不是最短时直接指出，并给出更小闭环方案。
3. 遇到问题追根因，不用补丁掩盖状态、数据或流程错误。
4. 输出只保留会改变决策的信息。

## 工程规范

- MVP 是 Next.js 全栈单体，不引入 Go、Redis、MQ、Worker、WebSocket 或服务端 PDF 渲染。
- 课程状态是产品主线，任何 API 修改状态都必须能回答“失败后如何恢复”。
- 图片是核心资产，`structured_lesson` 不保存图片 URL、prompt 或状态，统一由 `course_images` 管理。
- 能复用已成功图片时不重复消耗 AI 成本。
- 有明确业务行为的代码先写测试，再实现。
- UI 首屏直接进入可用产品，不做营销落地页。
- PDF 页面和 Preview 共用课程内容组件；PDF 永远隐藏答案和操作区。

## 前端模块开发流程

- 需求文档优先，线稿和设计稿只作为参考，不作为最终需求来源。
- 每次只讨论一个模块，不随意推进到下一个模块。
- 不清楚的产品、交互、数据、验收细节必须先提问确认。
- 对每个关键实现选择，需要给出建议和原因，再等待确认。
- 模块流程固定为：讨论细节 -> 写入模块文档 -> 用户确认文档 -> 开发 -> 自测 -> 用户验收。
- 模块文档统一放在 `docs/frontend/` 下。
- 每个模块结束后，在对应文档记录实现状态、验证命令和提交号。
- 新任务开始时，先读 `docs/frontend/README.md`，再读目标模块文档，避免依赖聊天记录。
- MVP 当前只考虑 Web 端，不做移动端 / App 端专项适配，除非用户重新确认范围。
